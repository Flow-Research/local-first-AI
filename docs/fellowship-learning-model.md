# Fellowship Learning Model

This project is organized so contributors learn by doing. AI can assist, but it should not replace human understanding.

## Learning Approach

Each week has four connected outputs:

| Output | Purpose | Human Responsibility |
|---|---|---|
| Research / learning | Understand one idea, paper, or technical source | Read, summarize, question, and connect it to the project |
| Design outcome | Turn learning into a design choice | Explain options, choose one, and name the tradeoff |
| Build outcome | Implement the smallest useful artifact | Build or inspect the work and understand how it behaves |
| Verification evidence | Prove the work is real | Run tests, review outputs, inspect screenshots, or record benchmark results |

## How AI Fits

AI is a study partner and drafting assistant.

AI can:

- Explain difficult papers.
- Generate draft summaries.
- Suggest architecture options.
- Help write first versions of code.
- Help review the repo for missing pieces.

AI should not:

- Replace reading.
- Replace human judgment.
- Commit unexplained code.
- Hide uncertainty.
- Become the only source of evidence.

## Fellowship Review Questions

Every contributor should be able to answer:

- What did I learn this week?
- What did I design from that learning?
- What did I build?
- How did I verify it?
- What did AI help with?
- What did I personally decide or inspect?

## Progression

The project moves from simple to complex:

1. Local-first foundation.
2. Local storage.
3. Local AI context.
4. Demo and evidence.
5. Constraints and benchmarks.
6. Sync thinking.
7. Privacy and security.
8. Device and local network work.
9. Agent workflow.
10. Evaluation.
11. Polish.
12. Final review.

This progression gives contributors enough structure to learn without being overwhelmed.
